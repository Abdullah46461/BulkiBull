# Email auth в MVP требует подтверждения почты до входа и использует API_PUBLIC_URL для ссылок

В MVP email-авторизация должна подтверждать владение почтой до первого входа. Регистрация не создает мгновенно активную сессию: пользователь сначала получает verification link, подтверждает адрес, и только потом может войти по email и паролю.

## Почему так

- это закрывает базовую проверку владения email без усложнения auth-сценария;
- логика понятна пользователю: сначала подтвердить почту, потом входить;
- сервер может безопасно инвалидировать и продлевать сессии независимо от клиента;
- `API_PUBLIC_URL` делает ссылку подтверждения управляемой через env для local, staging и production.

## Как это устроено

- `POST /auth/register` создает неподтвержденного пользователя и одноразовый verification token;
- `POST /auth/login` не пускает в аккаунт, пока `emailVerifiedAt` не заполнен;
- `GET /auth/verify-email` подтверждает адрес и помечает остальные незавершенные токены как использованные;
- сессии хранятся на сервере как hash opaque-token, а клиент передает `Authorization: Bearer <token>`;
- verification link всегда строится от `API_PUBLIC_URL`, а не из `localhost` по умолчанию.

## Практическое следствие

- для local web на одном компьютере `API_PUBLIC_URL` может быть `http://localhost:3000`;
- для проверки письма на телефоне или другом устройстве `API_PUBLIC_URL` должен указывать на адрес, достижимый получателем;
- для production нужен публичный `https`-домен API.
